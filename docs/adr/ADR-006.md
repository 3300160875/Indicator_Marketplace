# ADR-006: Database-authoritative quota reservation and settlement

- Status: accepted
- Date: 2026-06-25

## Context
Download quota reservation/settlement must be safe under concurrent requests and retryable database errors.

## Decision
Use MariaDB as the authoritative store for quota reservation and settlement. Use transactions, unique keys and row-level locking. Retry transient deadlocks/lock waits with bounded backoff.

## Consequences
- Quota mutations must go through `QuotaService`.
- Reservation identifiers must be unique and idempotent.
- Deadlock and lock wait errors are expected operational cases and must not be treated as successful quota changes.

## Verification
- `SELECT ... FOR UPDATE` blocked a second connection and preserved the row when the second update timed out.
- Duplicate unique-key insert returned `ERROR 1062`.
- Two-connection conflicting update produced `ERROR 1213 (40001)` deadlock for one transaction while the other committed.
