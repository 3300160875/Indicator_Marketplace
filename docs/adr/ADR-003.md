# ADR-003: Self-build entitlement engine instead of EDD All Access

- Status: accepted
- Date: 2026-06-25

## Context
The platform requires entitlement rules beyond basic EDD order storage. SR-006 runtime evidence proves EDD can provide order completion, order item IDs and full/partial refund data, while MariaDB can enforce quota reservation/settlement with transactions and unique keys.

## Decision
Self-build the entitlement engine instead of relying on EDD All Access. Entitlement state is derived from normalized EDD order/refund inputs and enforced by project-owned services/tables.

## Consequences
- Entitlement code must consume normalized data from `EddOrderAdapter`, not query EDD internals directly.
- No paid EDD entitlement plugin is approved by this ADR.
- Refund handling must reverse or reduce derived entitlement according to EDD refund order/item data.
- Quota reservation/settlement must stay database-authoritative with idempotency keys and bounded retry for deadlocks/lock waits.

## Verification
- MariaDB transaction/unique-key/deadlock behavior was verified for quota-style consistency needs.
- Runtime EDD spike produced full refund data with refund order/item total `-12.340000000` and sale status `refunded`.
- Runtime EDD spike produced partial item refund data with refund item total `-3.000000000` and sale status `partially_refunded`.
- Evidence: `docs/evidence/SR-006/runtime-edd-spike-summary.md`.
