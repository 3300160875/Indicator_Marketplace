# ADR-002: Use EDD Core as order and customer source of truth

- Status: accepted
- Date: 2026-06-25

## Context
Composer locks EDD 3.6.9, and SR-006 runtime spike installed disposable WordPress, activated EDD, completed orders and created full/partial refunds through public EDD APIs.

## Decision
Use EDD Core as the order and customer source of truth. Future business modules must read EDD order/customer state through an adapter boundary and must not duplicate order ownership in custom tables.

## Consequences
- `EddOrderAdapter` can consume EDD public APIs and hooks for completion/refund synchronization.
- Custom entitlement and quota tables store derived rights and usage, not authoritative order/customer state.
- Manual payment review must still update EDD order state before entitlements are granted.

## Verification
- `docs/spikes/SR-006/runtime-edd-spike.php` installed disposable WordPress, activated EDD and completed throwaway orders.
- First `edd_update_order_status($order_id, 'complete')` returned `true`; duplicate complete returned `false`.
- Runtime hooks observed: `edd_update_payment_status`, `edd_pre_complete_purchase`, `edd_complete_download_purchase`, `edd_complete_purchase`, `edd_after_payment_actions`, `edd_after_order_actions` and `edd_refund_order`.
- Evidence: `docs/evidence/SR-006/runtime-edd-spike-summary.md`.
